## 0. TL;DR

1. 이 코퍼스는 **하나의 전통(헬레니즘/중세) 점성술 체계로 통일**돼 있고, `이론 규칙 → 적용 예시 → 실제 상담기록`의 3계층이 갖춰져 있어 그대로 파이프라인화하기 좋다.
2. 깊이가 **현재 truefuture 정적 지식을 압도**한다. 특히 Notion의 **하우스별(2~12)·주제별 판단 파일**은 "키워드"가 아니라 **if→then 규칙 단위**라, 차트 요소로 키 검색해 프롬프트에 주입하기 최적이다.
3. "LLM 위키"는 **유료 서비스가 전혀 필요 없다.** 이 도메인은 검색 키가 결정론적(차트 계산이 "어떤 행성·사인·하우스·디그니티·랏을 볼지"를 이미 알려줌)이므로, **임베딩/벡터DB 없이 구조화 규칙 KB + 키 룩업**만으로 충분하다. 이게 가장 정확하고 가장 무료다. 산문 이론은 선택적으로 Supabase **pgvector**(이미 보유한 스택, 무료) RAG로 보강.
4. **기법은 공개 지식**이므로 자유롭게 코드화한다. 출처·저작권 표시는 넣지 않는다. (지식 모듈의 모든 규칙 텍스트는 자체 표현으로 재서술되어 작성됨.)
5. **풀이 문서는 내용/기법 참고용이 아니다.** 본인이 쓴 글이므로 **말투(화법)와 해석 순서**만 추출해 프롬프트의 화법 지시로 사용했다. 개인정보가 든 풀이/상담 본문 자체는 지식베이스에 넣지 않는다.
6. **자격증명 파일은 삭제 완료** (`Quick Note/개인 정보` 페이지 + `ID & Password` — 2026-06-23 제거). 나머지 유용한 파일(개인의 성격 및 성향 등)은 보존.
7. **파인튜닝은 현 단계 비권장** — 검증 사례 수십 건 미만 + 비표준 용어. **키-룩업 KB + 규칙 테이블**이 정답(갱신·일관성·무료).

## 3. "LLM 위키" — LLM이 읽기 좋은 지식베이스, 무료로 적용하는 법

### 3-1. 개념 정리: 우리에게 맞는 건 "벡터 RAG"가 아니라 "키 룩업 KB"

"LLM이 참고할 문서를 읽기 좋게 제공"은 보통 RAG(임베딩→벡터DB→유사도 검색)를 떠올리지만, **점성술은 검색 키가 결정론적**이라는 결정적 특성이 있다.

> 차트 계산이 끝나면 "이 사람은 **태양=염소자리 10하우스, 디그니티 중립, 낮차트**, 결혼랏 로드=화성@7하우스 손상…" 처럼 **무엇을 찾아봐야 하는지가 이미 확정**된다. "비슷한 문서를 찾는" 유사도 검색이 필요 없다. **정확한 키로 사전을 펼치면 된다.**

따라서 최적·최저비용 구조는:

```
[차트 엔진(코드)]  →  정규화된 사실 토큰 산출
   예) sun.sign=Capricorn, sun.house=10, sun.dignity=peregrine, sect=day,
       marriageLotLord=Mars, marriageLotLord.house=7, marriageLotLord.dignity=detriment, ...
        │
        ▼
[규칙 KB(구조화 JSON/markdown)]  →  토큰을 키로 매칭되는 해석 규칙 조회
   예) house10["직업·명예"], planetCombo["Mars+Saturn"]["직업"],
       rule["marriageLot_afflicted_by_malefic"] → "이혼 위험↑(단 7H 목성 카운터 체크)"
        │
        ▼
[프롬프트 주입]  →  매칭된 규칙 텍스트를 system/user 프롬프트에 사실로 첨부
        │
        ▼
[Gemini]  →  규칙을 "추론"하지 않고 그대로 적용해 자연어 서술만 생성
```

이 구조의 장점: **결정론적·일관적**(같은 차트=같은 근거), **환각 차단**(LLM이 점성술을 지어내지 않음), **완전 무료**(파일 + 코드뿐, 외부 서비스 0), **출처 추적·갱신 용이**(KB만 고치면 전 운세에 반영).

### 3-2. 무료 스택 (구조 이용 비용 = 0)

| 레이어 | 무료 선택지 | 비고 |
|---|---|---|
| **지식 저장** | 레포 내 `knowledge/*.json` 또는 `*.md` (git 버전관리) | Notion KB가 이미 80% 완성형. 서비스 비용 0 |
| **검색** | **키 룩업**(임베딩·벡터DB 불필요) | Deno 엣지 함수 안에서 객체 조회. 0 |
| **(선택) 산문 RAG** | Supabase **pgvector** (이미 보유 스택, 무료 확장) | "12 sign 138KB", 재물론 등 키로 못 쪼개는 산문 이론용 |
| **(선택) 임베딩 생성** | 빌드시 1회 오픈모델로 생성 / 무료 티어 | 키 룩업만 쓰면 임베딩 자체가 불필요 → 완전 무료 경로 |
| **공개용 LLM-readable 문서** | `llms.txt` 오픈 표준 (사이트 루트 마크다운) | 외부 LLM/크롤러가 진짜미래를 잘 이해하게 하는 무료 표준. 내부 해석 주입과는 별개 |

**결론: "무료여야 한다"는 제약은 전혀 문제 없음.** 1차 구조(키 룩업 KB)는 외부 서비스가 0이고, pgvector조차 이미 쓰는 Supabase의 무료 확장이다.

### 3-3. 현재 코드와의 접점 (어디를 바꾸나)

- **대체 대상(현재 빈약)**: `utils/dignityCalculator.ts`의 `HOUSE_MEANING`(하우스당 1~2단어), `utils/chartFormatter.ts`의 `SIGN_KEYWORDS`(4~5단어), `OUTER_PLANET_KEYWORDS`.
- **주입 지점(이미 존재)**: `chartFormatter.ts`의 `generate*UserPrompt()`, 그리고 `index.ts`가 `DIGNITY_SECTION_HEADER + neo4jContext`를 붙이는 흐름(`index.ts:1052`). 여기에 "KB 룩업 결과" 섹션을 같은 방식으로 추가하면 된다.
- **이미 있는 룰테이블 패턴**: `geminiPrompts.ts`의 `getLifetimePrompt_MoneyCareer()`에 이미 "수성+토성=연구직" 같은 미니 조합표가 있다(`:211~219`). 코퍼스의 행성쌍 15조합 사전으로 이걸 **무손실 확장**하면 된다.
- **계산은 이미 됨**: 디그니티·섹트·헤이즈·시너스트리·프로펙션·솔라리턴·어스펙트·랏이 모두 코드 산출 → 룩업 키로 바로 사용 가능.

## 4. 단계별 실행 로드맵

### Phase 1 — 정적 키 룩업 KB (무료·고ROI, 외부 의존 0)
1. **약어·동의어 정규화 사전** 구축 (선결작업): QS/WS, LoF/PoS/PoF/PoA, 컴버스트, 어버전, 연주, 달퍼 등. 구술체 코퍼스라 필수.

   - 스키마 예: `{ house: 10, topic: "직업·명예", rules: [{ when: {planet:"Sun", state:"angular"}, then: "사회적 지위 상승…" }] }`

### Phase 2 — 동적 토큰 연동
5. 차트엔진이 `marriageLotLord=Mars@7H/detriment` 같은 **정규화 토큰**을 산출하도록 포맷터 확장.

7. 궁합·시기추론에 **few-shot**(Compatibility_Analysis, 풀이) 1~2건 삽입 — 단 개인정보 **합성치환** 후.
